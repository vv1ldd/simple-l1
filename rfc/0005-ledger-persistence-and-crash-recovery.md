# RFC-0005: Ledger Lineage Persistence & Crash Recovery

**Статус:** Черновик (Draft)  
**Автор:** Команда Simple-L1  
**Дата:** 13 мая 2026 г.  

---

## 1. Аннотация (Abstract)

Этот RFC формализует **Phase 2D: Persistence Layer** протокола Simple-L1. Мы решаем проблему эфемерности состояния, вводя спецификацию постоянного хранилища (Persistent Block Store) и протокол восстановления после сбоев (Crash Recovery). Документ описывает структуру аппенд-онли журнала блоков (Append-Only Block Log) и алгоритм **Deterministic Replay from Genesis**, гарантирующий, что перезагруженная «холодная» нода восстановит абсолютно идентичное состояние сети и `state_root`.

---

## 2. Мотивация (Motivation)

Текущий рантайм Simple-L1 полностью консистентен в оперативной памяти, но уязвим перед перезагрузкой процессов или перебоями питания. 
Для долгосрочной выживаемости распределенной сети требуется:
1.  **Долговечность (Durability):** Совершенные мутации должны фиксироваться на энергонезависимом носителе (HDD/SSD) перед отправкой подтверждения клиенту.
2.  **Линейное доказательство (Lineage):** Цепочка блоков должна образовывать хэш-связную последовательность, защищенную от несанкционированного изменения (tamper-proof).
3.  **Сходимость при восстановлении:** Процесс перезапуска ноды обязан быть детерминированным автоматом переигрывания (Replay Machine), переводящим систему из нулевого состояния (Genesis) в актуальное.

---

## 3. Спецификация Append-Only Журнала (Ledger Log)

Базовое хранилище Simple-L1 представляет собой упорядоченный файл-журнал (Flat File Database), где записи добавляются строго в конец.

### 3.1 Структура Записи (Log Record Frame)
Каждый фрейм в файле содержит сериализованный Borsh-блок и служебные маркеры длины:

```rust
struct LogRecordFrame {
    magic_bytes: [u8; 4],            // Магическое число "SL1L" (Simple L1 Ledger)
    block_length: u32,               // Размер сериализованного блока
    serialized_block: Vec<u8>,       // Borsh([BlockHeader || BlockBody])
    checksum: [u8; 32],              // BLAKE3 хэш для детекции физического повреждения диска
}
```

---

## 4. Протокол Хэш-Связности (Tamper-Proof Lineage)

Связность блоков в журнале проверяется рекурсивно через поле `parent_hash`:

$$\text{Block}_{n}.\text{header}.\text{parent\_hash} == \text{BLAKE3}(\text{Borsh}(\text{Block}_{n-1}.\text{header}))$$

Если злоумышленник попытается модифицировать хотя бы один бит в теле `Block_{k}` на диске:
1.  Изменится `tx_root` в заголовке `Block_{k}`.
2.  Изменится `BlockHash` этого блока.
3.  Проверка Lineage для `Block_{k+1}` сломается (ссылка на родителя станет неверной).
4.  Каскадный сбой хэшей сделает все последующие блоки невалидными.

## 5. Алгоритм Восстановления Состояния (Deterministic Recovery)

При старте ноды («Холодный старт») запускается следующая процедура:

1.  **Инициализация Состояния:**
    *   `global_state = Map::new()`
    *   `last_applied_block_hash = [0; 32]`
    *   `height = 0`
2.  **Чтение файла журнала:**
    *   Открыть `ledger.log` для последовательного чтения.
3.  **Итеративное переигрывание (Replay Loop) для каждого блока:**
    *   Считать фрейм, проверить `magic_bytes` и `checksum`. В случае несовпадения — остановить запуск (ошибка диска).
    *   Распаковать `block_header` и `block_body`.
    *   **Проверить связность:** Если `height > 0`, убедиться, что `block_header.parent_hash == last_applied_block_hash`.
    *   **Восстановить Execution Batch:** Прогнать все транзакции из тела через рантайм валидации (см. RFC-0004).
    *   **Сверить контрольную точку:** Вычислить результирующий `state_root` после выполнения батча и сравнить его с `block_header.state_root`.
    *   **Обновить указатель:** `last_applied_block_hash = BLAKE3(Borsh(block_header))`, инкрементировать `height`.
4.  **Ввод в строй (Ready State):**
    *   Если весь файл успешно проигран, нода объявляет себя синхронизированной и готова к обработке новых транзакций из сети.

## 6. Изоляция Записи (Atomic Flush)

Для исключения порчи данных при внезапном отключении питания во время записи блока, нода обязана использовать атомарный системный призыв `fsync` (или аналог в Rust: `file.sync_all()`).

Алгоритм записи:
1.  Сериализовать блок в байты.
2.  Дописать фрейм в конец файла `ledger.log`.
3.  Вызвать `fsync()`. 
4.  Только после успешного ответа ОС транзакция считается **Finalized** на диске и сообщается клиенту.
