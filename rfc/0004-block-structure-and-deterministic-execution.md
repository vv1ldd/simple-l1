# RFC-0004: Block Structure & Deterministic Batch Execution

**Статус:** Черновик (Draft)  
**Автор:** Команда Simple-L1  
**Дата:** 13 мая 2026 г.  

---

## 1. Аннотация (Abstract)

Этот RFC формализует **Phase 2: State Coherence Layer** протокола Simple-L1. Мы переходим от валидации единичных approved intents к моделированию распределенного реестра. Документ утверждает строгую структуру блока (**Block Header & Body**), описывает конструирование детерминированного дерева Merkle execution records и фиксирует математические правила пакетного применения авторизованных транзакций (**Batch Execution**) для вычисления инвариантного корня состояния (**State Root**).

---

## 2. Мотивация (Motivation)

Для достижения консенсуса между независимыми узлами сети недостаточно верифицировать валидность отдельной транзакции. Необходимо гарантировать, что:
1.  **Порядок имеет значение:** Любая последовательность execution records $[Exec_1, Exec_2, ... Exec_n]$, примененная к начальному состоянию $S_0$, обязана дать абсолютно идентичное конечное состояние $S_1$ на любой ноде в мире.
2.  **Компактное доказательство:** Состояние всей сети в конкретный момент должно выражаться одним 32-байтовым хэшем (`state_root`), проверяемым легкими клиентами.
3.  **Изоляция времени и математики:** Любые источники неопределенности (дрейф локальных часов, числа с плавающей запятой) должны быть изгнаны из ядра исполнения.

## 3. Структура Блока (Block Schema)

Блок в Simple-L1 состоит из заголовка (**Header**) и тела (**Body**), сериализуемых через Borsh.

### 3.1 Спецификация Block Header (Borsh)
Заголовок содержит метаданные и криптографические обязательства (commitments).

| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `parent_hash` | `[u8; 32]` | Хэш заголовка предыдущего блока (Lineage). |
| `height` | `u64` | Порядковый номер блока в цепи (Genesis = 0). |
| `timestamp` | `u64` | Абсолютное UNIX-время создания блока (секунды). |
| `tx_root` | `[u8; 32]` | Корень бинарного дерева Merkle execution records блока. |
| `state_root` | `[u8; 32]` | Хэш глобального состояния сети ПОСЛЕ выполнения блока. |
| `proposer` | `[u8; 32]` | Хэш публичного ключа валидатора, собравшего блок. |

### 3.2 Спецификация Block Body (Borsh)
Тело содержит упорядоченный вектор execution records.
```rust
struct ExecutionEnvelope {
    approval: IntentApprovalEnvelope, // См. RFC-0003
    authorization: Authorization,     // Authority lineage accepted for execution
    transaction: Transaction,         // State mutation produced after validation
}

struct BlockBody {
    executions: Vec<ExecutionEnvelope>,
}
```

### 3.3 Вычисление Block Hash
Уникальный ID блока:
$$\text{BlockHash} = \text{BLAKE3}(\text{Borsh}(\text{BlockHeader}))$$

## 4. Конвейер Хэширования Execution Records: Merkle Root

Для формирования `tx_root` используется стандартное сбалансированное бинарное дерево хэшей (Binary Merkle Tree) от `ExecutionHash` из тела блока.

1.  **Листья (Leaves):** Для каждого execution record вычисляется $L_i = \text{BLAKE3}(\text{Borsh}(ExecutionEnvelope_i))$.
2.  **Попарное хэширование:**
    *   Если количество листьев нечетно, последний лист дублируется.
    *   Родительский узел вычисляется как $\text{BLAKE3}(Left || Right)$.
3.  Операция повторяется рекурсивно, пока не останется один корень — `tx_root`.

## 5. Детерминированное Глобальное Состояние (State Commitments)

Глобальное состояние (Accounts State) Simple-L1 логически представляет собой Key-Value хранилище:
`Address (20 bytes) -> AccountEntry { balance: u128, nonce: u64 }`

### 5.1 Вычисление State Root
Для обеспечения воспроизводимого хэширования без использования сложных деревьев Trie на ранней стадии (Embryonic Runtime), Simple-L1 утверждает **Sorted State Serialization**:

1.  Все измененные или активные аккаунты сортируются лексикографически по их байтовому `Address`.
2.  Каждая запись сериализуется в бинарный поток: `[Address (20 bytes) || Balance (16 bytes) || Nonce (8 bytes)]`.
3.  Результирующий сквозной байтовый поток хэшируется:
    $$\text{StateRoot} = \text{BLAKE3}(\text{ConcatenatedSortedAccountBytes})$$

Любое расхождение в балансе хотя бы на 1 базовую единицу или отличие в `nonce` хотя бы одного аккаунта породит абсолютно другой `state_root`, приводя к мгновенному отторжению блока консенсусом.

## 6. Законы Batch Execution (Deterministic Invariants)

При получении нового блока, валидатор запускает конвейер последовательных переходов:

$$State_{current} \xrightarrow{Exec_1} State_a \xrightarrow{Exec_2} State_b ... \xrightarrow{Exec_n} State_{next}$$

Для предотвращения фатальных расхождений детерминизма (Determinism Hazards) вводятся железные правила:

### 6.1 Целочисленная математика (No Float)
Все операции с балансами и комиссиями осуществляются ИСКЛЮЧИТЕЛЬНО в целочисленном формате (`u128` для балансов). Использование типов с плавающей точкой (`f32`/`f64`) в рантайме сети ЗАПРЕЩЕНО.

### 6.2 Инвариант Времени (Consensus Timestamp)
Поле `timestamp` внутри заголовка блока является единственным источником времени для всех транзакций внутри этого блока. Валидаторы игнорируют свои системные часы при исполнении логики транзакций (например, `expires_at`), сверяя её только с `header.timestamp`.

### 6.3 Монотонность времени
Валидный блок обязан иметь `timestamp` строго больше, чем у родительского блока:
$$\text{Block}_{n}.\text{timestamp} > \text{Block}_{n-1}.\text{timestamp}$$

## 7. Жизненный Цикл Верификации Блока на Ноде

Когда нода получает предложенный блок:
1.  **Проверка Lineage:** Убедиться, что `parent_hash` совпадает с хэшем локального родительского блока.
2.  **Проверка Времени:** Убедиться в монотонности `timestamp`.
3.  **Реконструкция Tx Root:** Собрать Merkle Tree от execution records в теле и сверить полученный корень с `header.tx_root`.
4.  **Batch Execution:**
    *   Установить точку отката состояния (Snapshot).
    *   Последовательно проверить `IntentApproval -> Authorization -> Transaction` и применить результат. В случае сбоя хотя бы одного execution record — блок невалиден, состояние откатывается.
5.  **Верификация State Root:** Вычислить `StateRoot` получившегося состояния и сравнить с `header.state_root`.
6.  **Commit:** При успешном совпадении всех инвариантов блок фиксируется в локальной цепочке, мутация состояния становится необратимой.
