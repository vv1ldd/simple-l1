# RFC-0011: Identity Kernel & Capability Resolution

**Status:** Frozen v1
**Date:** 26 May 2026

---

## 1. Abstract

Simple-L1 identity is a passkey-backed deterministic identity model, not a passkey-as-account model.

The kernel separates:

* `entity_l1_address` — who exists.
* `key_l1_address` — which passkey proves control.
* `capability_grant` — what the entity/key may do.

This RFC freezes the v1 boundary between identity, authentication, and authorization.

---

## 2. Rule #0

`entity_l1_address` MUST NOT be derived from passkey public key material.

Passkeys authenticate and sign. They do not define the account.

---

## 3. Address Classes

### 3.1 Entity Address

`sl1e_` identifies the stable account/entity.

Properties:

* Stable across key rotation.
* May own balances, profiles, grants, policies, and settlement projections.
* Is the address used by intents, commerce, marketplace, partner records, and cross-domain identity proof.
* Is generated by the identity kernel, not by a passkey hash.

Canonical pattern:

```text
^sl1e_[a-f0-9]{39}$
```

### 3.2 Key Address

`sl1_` identifies passkey-derived proof material.

Properties:

* Derived from one passkey public key.
* May be registered, revoked, rotated, or scoped.
* Proves control during authentication/signing.
* MUST NOT be used as account identity.

Canonical pattern:

```text
^sl1_[a-f0-9]{40}$
```

---

## 4. Identity Kernel API

The kernel exposes two strict lanes:

* Entity lane: create, normalize, assert, resolve `sl1e_`.
* Key lane: derive, normalize, assert `sl1_`.

No API may silently coerce a key address into an entity address.

---

## 5. Authorization Contract

Authorization is not a role system.

Roles describe key or operational posture. Roles never decide permission.

Permission is decided only by explicit grants:

```text
simple_l1_capability_grants(
  entity_l1_address,
  key_l1_address?,
  capability,
  scope,
  policy,
  status,
  expires_at?
)
```

---

## 6. CRE v1

Capability Resolution Engine v1 is a deterministic function:

```text
decide(entity_l1_address, proof_key_l1_address?, capability, scope, context) -> decision
```

Inputs:

* `entity_l1_address`: required `sl1e_`.
* `proof_key_l1_address`: optional `sl1_`.
* `capability`: exact string, no wildcard expansion.
* `scope`: exact string, no wildcard expansion.
* `context`: opaque metadata, non-decisional in v1.

Output:

```json
{
  "decision": "allow|deny|quorum|approval",
  "matched_grants": [],
  "reason_codes": [],
  "confidence": 1
}
```

Policy precedence:

```text
deny > require_quorum > require_approval > allow
```

No graph traversal, cache state, role lookup, social inference, or execution side effect is part of CRE v1.

---

## 7. Frozen Boundaries

Identity answers: who exists?

Authentication answers: which key proved control?

Authorization answers: which explicit grants match?

Execution answers: what state transition happens after authorization?

These layers MUST remain separate.
