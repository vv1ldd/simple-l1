# RFC-0002: Intent & WebAuthn Serialization Schema

**Статус:** Черновик (Draft)  
**Автор:** Команда Simple-L1  
**Дата:** 13 мая 2026 г.  

---

## 1. Аннотация (Abstract)

Этот RFC определяет стандарт сериализации намерений (Intents) и протокол их аппаратной подписи через **WebAuthn (FIDO2)**. Мы описываем, как структура намерения превращается в криптографический `challenge`, как клиентское устройство подписывает его и как валидатор сети Simple-L1 детерминированно проверяет эту сложную подпись.

---

## 2. Проблема: Как подписывать L1 данные на телефоне?

Обычные аппаратные анклавы (Apple Secure Enclave, Android Keystore) через браузерный API (WebAuthn) **никогда не подписывают сырые бинарные данные напрямую**. Из соображений безопасности они подписывают структуру, которая включает в себя:
1.  `authenticatorData` — байты от самого чипа (флаги User Presence, счетчик подписей).
2.  `clientDataJSON` — JSON-объект с хэшем исходного `challenge`, источником (origin) и типом запроса.

Подпись вычисляется над конкатенацией:
$$\text{Signature} = \text{ECDSA}_{SK}(\text{authenticatorData} \ || \ \text{SHA-256}(\text{clientDataJSON}))$$

**Задача:** Спроектировать транзакцию так, чтобы исходное намерение (Intent) лежало внутри `challenge` в `clientDataJSON`, обеспечивая криптографическую неразрывность.

---

## 3. Спецификация Намерения (Intent Layout)

Намерение — это чистая бизнес-логика того, что пользователь хочет сделать.

### 3.1 Бинарный формат Intent (Borsh/Canonical Binary)
Для детерминированной сериализации на клиенте и валидаторе мы используем строгий бинарный формат (предлагается **Borsh** или **ProtoBuf/Canonical**).

| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `chain_id` | `u32` | Идентификатор сети Simple-L1 для защиты от replay между тестнетом/мейннетом. |
| `nonce` | `u64` | Порядковый номер транзакции для этого аккаунта. |
| `valid_until` | `u64` | Номер слота/блока, после которого транзакция недействительна. |
| `fee` | `u64` | Максимальная комиссия в базовых единицах (сатоши-эквивалент). |
| `action_type` | `u8` | Код действия: `1` = Transfer, `2` = RegisterKey, и т.д. |
| `payload` | `bytes` | Аргументы действия (например, адрес получателя + сумма). |

## 4. Конвейер Подписи (Signing Pipeline)

Когда пользователь инициирует действие в веб-интерфейсе:

1.  **Формирование Intent**: Клиент собирает структуру данных и сериализует её в байты $I_{bytes}$.
2.  **Хэширование Intent**:
    $$H_{intent} = \text{BLAKE3}(I_{bytes})$$
3.  **WebAuthn Challenge**: Полученные 32 байта хэша кодируются в URL-Safe Base64 (без padding) и передаются в API устройства:
    ```javascript
    const challenge = base64url.encode(H_intent);
    const assertion = await navigator.credentials.get({
      publicKey: {
        challenge: challenge,
        rpId: "simple-l1.network",
        // ...остальные параметры
      }
    });
    ```
4.  **Упаковка Транзакции**: После того как пользователь приложил FaceID/TouchID, на L1-ноду отправляется пакет, содержащий:
    *   Исходное намерение ($I_{bytes}$)
    *   `authenticatorData` (байты от чипа)
    *   `clientDataJSON` (текст JSON)
    *   Подпись `(r, s)` (ASN.1 DER или сырые 64 байта)

## 5. Алгоритм Верификации на Ноде (Validator Consensus Loop)

Валидатор получает транзакцию и выполняет следующие строгие шаги проверки:

### 5.1 Валидация привязки Intent
1.  Парсит `clientDataJSON` (текстовый JSON).
2.  Извлекает поле `challenge` из JSON.
3.  Вычисляет $H_{intent\_calc} = \text{BLAKE3}(I_{bytes})$.
4.  Сравнивает: декодированный `challenge` **обязан в точности совпадать** с $H_{intent\_calc}$.
    *(Если не совпадает — транзакция отбрасывается, подпись подделана).*

### 5.2 Проверка контекста WebAuthn
1.  Проверяет флаг **User Presence (UP)** и **User Verification (UV)** в `authenticatorData`. Для Simple-L1 бит `UV` (пользователь биометрически верифицирован) должен быть равен `1`.
2.  Проверяет `origin` или `rpIdHash` в зависимости от политики безопасности сети.

### 5.3 Математическая Верификация ECDSA
1.  Вычисляет SHA-256 хэш от оригинальной текстовой строки `clientDataJSON`:
    $$H_{json} = \text{SHA-256}(clientDataJSON)$$
2.  Собирает итоговое подписанное сообщение:
    $$M_{signed} = \text{authenticatorData} \ || \ H_{json}$$
3.  Извлекает публичный ключ $PK$ аккаунта отправителя из состояния сети.
4.  Выполняет ECDSA верификацию подписи `(r, s)` на эллиптической кривой **secp256r1 (P-256)** над хэшем SHA-256 от сообщения $M_{signed}$:
    $$\text{Verify}_{PK}(\text{SHA-256}(M_{signed}), \ (r, s)) == \text{Valid}$$

## 6. Анализ Безопасности и Атак

### 6.1 Replay Атаки
Благодаря жесткой проверке `nonce` внутри `Intent`, одна и та же подпись не может быть исполнена дважды.
Защита от replay между сетями обеспечивается полем `chain_id`.

### 6.2 Модификация JSON (JSON Canonicalization)
Поскольку устройство подписывает конкретную байтовую строку `clientDataJSON`, которую оно само сгенерировало, валидатору не нужно заниматься нормализацией (canonicalization) JSON. Ему достаточно взять те самые байты, которые прислал клиент, вычислить их хэш и сравнить поле `challenge`. Это радикально упрощает логику валидатора и защищает от уязвимостей парсинга JSON.
